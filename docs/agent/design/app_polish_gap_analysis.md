# App 精致感缺口分析

## 用途

记录 2026-07-02 对当前桌面 app “不够精致”的只读审视结论。本文用于后续 UI/UX 收敛时判断优先级，不替代 `DESIGN.md` 的长期设计规范。

## 审视依据

- 当前开发态 Vite 预览：`http://localhost:7273/`。
- 实际观察：欢迎页、侧栏隐藏态、标题栏、空文件夹入口、主操作按钮层级。
- 代码依据：`apps/desktop/src/app/App.tsx`、`apps/desktop/src/styles.css`、`packages/editor-ui/src/components/WelcomeState.tsx`、`packages/editor-ui/src/components/DocumentBar.tsx`、`packages/editor-ui/src/components/MilkdownEditor.css`。
- 设计依据：`docs/agent/design/DESIGN.md` 中“克制、准确、本地优先、写作默认优先”的目标。

## 当前已成立的改进

应用已经不是早期的“功能散落”状态。当前代码里已经有欢迎页、可隐藏侧栏、文件/大纲切换、底部文档控制、窗口标题、资源返回和编辑器局部 loading 等结构。

因此，“不够精致”的主要问题不是缺少组件，而是这些组件还没有形成统一的视觉秩序和稳定的产品气质。

## 排名结论

| 排名 | 问题 | 置信度 | 依据 |
| --- | --- | --- | --- |
| 1 | 视觉层级过轻，导致界面像半透明的线框稿 | 高 | 全局 token 大量使用白、浅灰和低对比蓝；标题栏、侧栏、分割线、图标都弱化。 |
| 2 | 桌面外壳缺少稳定锚点 | 高 | 标题、侧栏入口、文档控制分散在顶部、左上角和侧栏底部；隐藏侧栏后左上入口像孤立浮层。 |
| 3 | 欢迎页构图偏松，首屏不像完整产品首页 | 高 | 欢迎模块居中漂浮，左侧/顶部留白巨大；“打开文件夹”在侧栏和主区域重复出现但没有形成任务流。 |
| 4 | 控件语言可用但不够系统化 | 高 | 30px 图标按钮、5px 圆角、浅 hover、局部阴影散落在多个 className 中，缺少统一的控件密度和状态层级。 |
| 5 | 写作画布有 Typora-like 基础，但仍偏默认文档渲染 | 中 | 860px 内容宽度、白底和 GitHub 风标题边线可用，但缺少更细的正文节奏、段落密度、代码块层级和编辑态反馈。 |
| 6 | 状态反馈还没有产品化 | 中 | 未保存主要靠标题星号；loading 和 toast 可见但像局部补丁，缺少稳定的“已保存/未保存/保存中/错误”状态系统。 |

## 证据与推断

### 1. 视觉层级过轻

证据：

- `apps/desktop/src/styles.css` 中默认背景、chrome、surface 都接近白色，边框是 `#eeeeee`，弱化文字是 `#999999`。
- `AppTitleBar` 使用 `text-[var(--theme-muted)]`，标题栏高度 34px，视觉存在感很低。
- 侧栏标题、图标按钮和搜索入口都使用低对比线性图标和透明背景。

推断：

当前界面在“克制”上做到了，但缺少主次对比。结果是用户能感到干净，却不容易感到“被设计过”。精致感需要更明确的层级：窗口 chrome、工作区、写作画布、临时浮层和状态反馈应有不同的重量。

### 2. 桌面外壳缺少稳定锚点

证据：

- 主标题在 `AppTitleBar` 居中渲染；侧栏隐藏入口是编辑区域左上角的绝对定位按钮；模式切换和设置位于侧栏底部的 `DocumentBar`。
- 当侧栏隐藏时，主界面只剩左上浮动按钮和顶部居中标题，缺少一个贯穿全局的工具/状态带。

推断：

这会让应用显得像多个局部控件拼起来，而不是一个完整桌面 app。尤其在空状态下，左上角侧栏按钮、顶部标题、中央欢迎内容之间没有强关系。

### 3. 欢迎页不像完整产品首页

证据：

- `WelcomeState` 是一个 `m-auto` 的 620px 模块，包含 M 标识、标题、三个按钮和一句本地保存说明。
- 当前首屏大面积空白，但没有利用空白形成工作区、最近文件、项目入口或品牌稳定感。

推断：

欢迎页现在是“入口集合”，不是“第一屏体验”。它功能正确，但缺少产品气质：用户第一眼看到的是一组按钮，而不是一个清晰、可信、可持续使用的写作环境。

### 4. 控件系统不够统一

证据：

- 图标按钮样式分别散落在 `sidebarHeaderIconButtonClassName`、`DocumentBar`、侧栏隐藏按钮和其它组件 className 中。
- 按钮、toast、loading、标题栏、搜索面板各自直接写尺寸、圆角、阴影和边框。

推断：

这些控件单看都不差，但整体会显得“临时拼装”。精致感通常来自统一的控件谱系：同一类按钮、浮层、搜索、状态提示要共享尺寸、边框、hover、active、focus、禁用和动效规则。

### 5. 写作画布偏默认渲染

证据：

- `MilkdownEditor.css` 把编辑内容限制在 `--theme-content-width: 860px`，并设置 36px 顶部 padding、30px 横向 padding。
- 标题使用大字号和底部分割线；blockquote、inline code、code block 有基本样式。

推断：

这已经接近 Typora-like 的基础，但还没有形成独特的正文节奏。标题分割线、代码块、引用、列表、AI ghost text 的视觉语言仍偏“Markdown 默认主题”。如果目标是长期写作工具，需要更精细地处理段落间距、标题密度、代码块操作、选区反馈和空段落占位。

### 6. 状态反馈没有形成系统

证据：

- 未保存状态在标题后追加 `*`。
- `EditorToast` 是顶部短暂浮层；`EditorLoadingState` 是编辑区域内居中卡片式 loading。
- `DocumentBar` 当前只承载模式切换和设置，没有稳定显示保存、同步、错误或当前文档状态。

推断：

用户可能认为“功能能用但不稳”。不是因为数据真的不安全，而是保存、错误、loading、AI 状态和模式状态没有被组织成一个一致、持续可见的反馈系统。

## 优先修正方向

1. 先做一轮视觉层级收敛：明确 chrome、sidebar、canvas、floating、status 五层的背景、边框、阴影、字号和对比度。
2. 重整桌面外壳：把侧栏入口、文件身份、未保存状态、模式切换、设置入口纳入同一套稳定结构，避免左上孤立浮层。
3. 重做欢迎页构图：保留简洁，但让首屏表达“本地写作工作区”，而不是只展示三个按钮。
4. 抽出控件基元：至少统一 icon button、primary/secondary button、toast、loading、search field、status indicator 的状态规则。
5. 精修正文主题：按长文写作场景打磨标题、段落、列表、引用、代码块、空文档占位和选区状态。
6. 建立状态系统：用明确但低打扰的方式展示已保存、未保存、保存中、错误、AI 可用/不可用，而不是分散在标题星号和短暂 toast 中。

## 不确定项

- 本次没有打开真实项目文件夹和长文档样本，因此文件树密度、长文档大纲和正文滚动体验还需要单独截图验证。
- 本次没有覆盖设置窗口、源码搜索面板和 AI 建议实际触发态，它们可能还有独立的精致感问题。
- 浏览器预览不能完全替代 Tauri 桌面窗口，最终视觉判断还应在 macOS 原生窗口下截图复核。
