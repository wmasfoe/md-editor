# 能力边界设计原则

用途：记录“核心能力收敛到所属领域，外部接入层只做适配和注入”的架构风格。后续设计编辑器能力、AI 接入、平台 runtime、外部服务 provider 或跨包接口时，用本文判断职责边界和测试拆分。

## 核心原则

当一个功能同时涉及外部接入和内部交互时，先判断真正的问题本体属于哪个领域，再把稳定能力放回该领域的核心模块。

- 核心模块负责领域语义、状态机、渲染规则、验收契约和可复用 API。
- 接入层负责调用外部系统、协议转换、错误归一化、权限和配置。
- UI 容器负责组合与传参，不应复制核心模块里的交互和文档模型规则。
- 测试按边界拆分，避免为了验证一个 provider 而启动完整编辑器，也避免为了验证编辑器行为而依赖真实网络或模型。

## AI 与编辑器的边界

AI 写作能力采用以下分工：

- `@md-editor/ai` 层：读取设置、请求大模型或本地模型、解析 JSON/文本结果、返回纯 suggestion 数据。
- editor-core 层：定义编辑器命令、feature 契约和跨模式稳定接口；不 re-export AI 包，不承载 provider/request parsing。
- editor-ui / Milkdown 层：负责 suggestion 的展示、接受、取消、失效、选区保持、Markdown 插入、换行跟随和可访问交互。
- desktop app 层：负责设置页、provider 配置、本地模型状态、错误提示和把 AI 结果注入编辑器能力。

因此，类似“AI tab 补全建议跨行显示错位”的问题，本体是编辑器 suggestion 渲染和交互语义，不应在 AI 请求层或桌面容器里硬算位置。AI 层只产出 `continuation` / `edit` 等结构化结果，编辑器层提供稳定的 suggestion preview 能力来消费它。

## 设计检查表

新增或修复功能时，先过一遍：

1. 这个行为的长期主人是谁？
2. 调用方是否在重复核心模块已经知道的状态？
3. 未来换 provider、换 runtime、换 UI 容器时，这个能力是否还能复用？
4. 能否分别测试“外部结果解析”和“核心行为执行”？
5. 当前方案是否把一次性修复写成了稳定能力，还是把稳定能力散落成一次性逻辑？

如果答案显示调用方正在硬算核心领域状态，应回到核心模块预留 API，再让调用方通过该 API 注入输入。

## 验证方式

推荐把验证拆成两层：

- Adapter/provider 测试：输入 mock 响应，验证请求体、超时、错误处理和结果解析。
- Core/editor 测试：输入纯领域数据，验证渲染、状态转移、接受/取消行为、换行和选区规则。

只有跨层集成风险较高时，再补端到端测试。这样可以让问题定位更快，也能避免 AI、网络、平台 runtime 的不稳定性污染编辑器能力验证。
