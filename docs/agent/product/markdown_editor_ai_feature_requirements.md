# Markdown Editor AI Feature Requirements

用途：记录 AI 功能的阶段性结论、隐私边界和首批实现范围，作为后续开发 AI 基础设施与显式续写能力的依据。

## 1. 背景

项目原始目标包含 Typora-like Markdown 编辑体验、MDX 组件能力，以及 AI 纠错和 AI 生成文本能力。现有技术方案已经把 AI 写作辅助放在后置增强项中，规划过 AI suggestion、显式续写、静默检查、OpenAI-compatible endpoint、本地模型、历史文章本地索引和 `.aiignore`。

当前讨论后，AI 功能的近期目标收敛为：

- 先做 AI 基础设施。
- 先做 AI 显式续写，体验接近 Cursor IDE 的 inline suggestion。
- 暂不做用户风格学习。
- 暂不做 AI 静默检查。

## 2. 核心决策

### 2.1 用户风格分析必须使用本地模型

用户风格学习会读取大量历史文章、草稿、个人表达习惯和潜在隐私内容，因此它不能默认走远程模型。

要求：

- 用户风格分析只能使用本地模型。
- 历史文章索引、风格画像和相关中间产物默认保存在本地。
- 后续支持 `.aiignore`，让用户排除不希望参与索引或风格分析的文件。
- 不允许把历史文章批量上传到远程模型做风格分析。

### 2.2 本地模型不随 App 打包

为了减少安装包体积，本地模型不进入默认安装包。

要求：

- 用户安装 App 后，可以在设置页面手动下载本地模型。
- 设置页需要展示本地模型状态，例如未下载、下载中、可用、下载失败。
- 模型下载必须是用户主动行为，不能静默自动下载。
- 首版可以先预留下载入口和状态模型，具体模型下载实现可以后续单独做。

### 2.3 AI 显式续写优先于静默检查

显式续写由用户主动触发，隐私边界更清晰，也不会干扰写作节奏。静默检查需要持续监听输入、截取上下文并显示轻提示，复杂度和打扰风险更高。

要求：

- 第一阶段优先实现显式续写。
- 静默检查默认关闭，并延后实现。
- 即使后续支持静默检查，也必须由用户主动开启。

## 3. 首批功能范围

### 3.1 AI 基础设施

第一批基础设施应该只覆盖后续功能必需的最小能力：

- AI 设置入口。
- AI 总开关。
- Provider 类型：
  - OpenAI-compatible endpoint。
  - 本地模型。
- OpenAI-compatible 配置：
  - endpoint / base URL。
  - model。
  - API Key。
- 本地模型配置：
  - 本地模型启用状态。
  - 模型下载入口。
  - 模型可用状态。
- 请求失败提示：
  - 未配置 provider。
  - API Key 缺失。
  - endpoint 不可达。
  - 本地模型未下载。
  - 生成超时或取消。

暂不要求：

- 多 provider 市场。
- 云端账号体系。
- 内置 API Key。
- 自动选择模型。
- 风格学习索引。
- 后台静默检查。

### 3.2 AI 显式续写

目标体验：用户在编辑器光标处触发 AI 续写，编辑器显示 inline ghost suggestion，用户按 `Tab` 接受，按 `Esc` 取消。

功能要求：

- 用户通过命令或快捷键主动触发续写。
- 续写请求只发送当前光标附近必要上下文。
- 返回结果以 ghost text / inline suggestion 展示，不立即写入 Markdown。
- `Tab` 接受建议并插入正文。
- `Esc` 取消建议，不修改正文。
- 用户继续输入时，当前建议应取消或失效，避免插入过期内容。
- 生成中应有轻量状态提示，但不使用阻断式全局 loading。
- 失败时用 toast 或局部提示说明原因，不污染文档内容。

首版建议只覆盖：

- 当前文档内的显式续写。
- 单条 suggestion。
- 非流式返回或简单流式展示二选一；优先选择实现更稳定的路径。

首版暂不覆盖：

- 多候选建议。
- 对整篇文章的重写。
- 自动续写。
- 用户风格学习。
- 跨文件上下文检索。

## 4. 隐私与数据边界

AI 功能必须把数据边界展示清楚，尤其是远程 provider 和本地模型之间的差异。

要求：

- 远程 provider 只能在用户配置并主动触发时请求。
- 显式续写只发送必要上下文，不发送整篇历史文章。
- 用户风格学习只能使用本地模型。
- 静默检查必须默认关闭。
- 本地模型下载必须由用户主动点击。
- 后续实现 `.aiignore` 前，不做历史文章风格学习。

## 5. 与现有架构的关系

AI 功能应作为内置 Feature 接入现有编辑器运行时，而不是一开始设计为第三方插件运行时。

建议边界：

- `@md-editor/ai` 定义 AI provider、请求解析和 AI 相关类型；调用方直接从该包导入。
- `editor-core` 只定义编辑器命令、Feature descriptor 和跨模式编辑器契约，不 re-export AI 包。
- desktop app 负责设置、provider 配置、请求执行、错误提示和快捷键绑定。
- `renderer-codemirror` 在单一 `EditorView` 中负责 suggestion 展示、接受、取消、失效、选区映射和 history；AI 层只注入纯 suggestion 数据。
- WYSIWYG 与源码模式共享同一份 request / suggestion 状态，不按模式建立第二套 ghost text 或撤销语义；具体可见性由 renderer 的 mode StateField 决定。

不要在第一版引入：

- 新依赖。
- 后台索引服务。
- 第三方插件运行时。
- 用户风格画像存储。
- 自动下载模型。

## 6. 后续验收口径

AI 基础设施完成时，应能验证：

- 设置页可以配置 AI provider。
- 设置保存后重启仍保留非敏感配置。
- API Key 的存储策略有明确说明。
- 本地模型状态能在设置页呈现。
- 未配置时触发 AI 续写会给出清晰提示。

AI 显式续写完成时，应能验证：

- 用户主动触发续写后出现 ghost suggestion。
- `Tab` 接受后 Markdown 正文被插入。
- `Esc` 取消后 Markdown 不变。
- 用户继续输入后旧 suggestion 不会被错误接受。
- 失败、取消、超时不会改变文档内容。
- 续写不会破坏当前 Markdown / MDX raw 保真策略。

## 7. 明确延后项

- AI 静默检查。
- 用户风格学习。
- 历史文章本地索引。
- `.aiignore` 规则执行。
- 本地模型下载器的完整实现。
- 跨文件上下文检索。
- 多模型路由。
- prompt 模板市场。
